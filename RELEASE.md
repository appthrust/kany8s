# Release Runbook

kany8s is shipped as a Cluster API provider consumed via `clusterctl` or
`cluster-api-operator`. Releases are cut by hand: a maintainer tags the
commit, pushes the tag, and GitHub Actions does the rest.

Always read this file before cutting a release. It encodes the few
non-obvious details that the GitHub UI will not tell you.

## Release artifacts

Every published tag (`vX.Y.Z`) produces:

- Container images on GHCR (linux/amd64 + linux/arm64 manifest list):
  - `ghcr.io/appthrust/kany8s/manager:vX.Y.Z`
  - `ghcr.io/appthrust/kany8s/eks-kubeconfig-rotator:vX.Y.Z`
  - `ghcr.io/appthrust/kany8s/eks-karpenter-bootstrapper:vX.Y.Z`
- Helm chart OCI artifacts on GHCR:
  - `oci://ghcr.io/appthrust/charts/eks-kubeconfig-rotator:X.Y.Z`
  - `oci://ghcr.io/appthrust/charts/eks-karpenter-bootstrapper:X.Y.Z`
- GitHub Release assets:
  - `metadata.yaml`
  - `infrastructure-components.yaml`
  - `control-plane-components.yaml`
  - `cluster-template.yaml`
  - `clusterctl-config.yaml`

All images, charts, and release assets are public. The GHCR packages must
be set to "Public" visibility by hand the first time they are pushed, see
the troubleshooting section below.

## Automation topology

Two workflows fire on every tag push:

1. `release.yml` — on `push: tags: ['v*']` and `workflow_dispatch`. Builds
   the three images, composes the multi-arch manifest list, then attaches
   the five provider assets to a GitHub Release. Release notes are
   auto-generated by `softprops/action-gh-release` (`generate_release_notes:
   true`) from merged PRs since the previous tag.
2. `release-helm.yml` — on the same trigger. Waits for `release.yml` to
   publish the multi-arch image manifests, rewrites each `charts/*/Chart.yaml`
   `version` + `appVersion` from the tag, then pushes both Helm charts to
   `oci://ghcr.io/appthrust/charts/`. Skips idempotently if the chart version
   is already published.

There is no PR-based automation. Merging to `main` only runs CI; tagging
is what produces a release.

## One-time setup

### Branch protection

Require the following status checks on `main`:

- `ci`
- `smoke-test`

This prevents broken commits from reaching a tag.

### Conventional commits (optional)

CI does not enforce Conventional Commits. GitHub's auto-generated release
notes list merged PRs verbatim, so a clean PR title ("Add foo", "Fix bar")
is enough. Use `feat:` / `fix:` / `BREAKING CHANGE:` prefixes only if you
want the notes to group commits by category.

## Cutting a release

1. Check that the tip of `main` is green on `ci` and `smoke-test`.
2. Decide the next version following SemVer. Pre-major (`0.x`) bumps:
   - Breaking change → bump minor (`0.2.0` → `0.3.0`)
   - Feature or fix → bump patch (`0.2.0` → `0.2.1`)
3. (Optional) Dry-run the build pipeline without pushing images:
   ```bash
   gh workflow run release.yml \
     --field tag=vX.Y.Z-rc0 \
     --field dry_run=true
   ```
4. Tag and push:
   ```bash
   git tag -a vX.Y.Z -m "vX.Y.Z"
   git push origin vX.Y.Z
   ```
   This fires `release.yml` and `release-helm.yml` in parallel. Watch
   both at `https://github.com/appthrust/kany8s/actions` until they
   finish green.
5. Open the GitHub Release created by the workflow, verify the five
   attached assets and the auto-generated notes, then click
   `Publish release` (if the workflow left it as a draft).
6. Smoke-install with the published assets:
   ```bash
   clusterctl init \
     --config clusterctl-config.yaml \
     --infrastructure kany8s \
     --control-plane kany8s
   ```

## Cutting the bootstrap release (first time only)

The very first release of a new minor line needs two extras because no
GHCR packages exist yet and there is no prior Release to anchor
`clusterctl-config.yaml`.

1. Tag and push as above.
2. Wait for `release.yml` to finish, then switch the GHCR packages to
   public visibility:
   ```bash
   for pkg in manager eks-kubeconfig-rotator eks-karpenter-bootstrapper; do
     gh api \
       --method PATCH \
       -H "Accept: application/vnd.github+json" \
       "/orgs/appthrust/packages/container/kany8s%2F${pkg}" \
       -f visibility=public
   done
   for chart in eks-kubeconfig-rotator eks-karpenter-bootstrapper; do
     gh api \
       --method PATCH \
       -H "Accept: application/vnd.github+json" \
       "/orgs/appthrust/packages/container/charts%2F${chart}" \
       -f visibility=public
   done
   ```
   (The same toggle exists in the GHCR UI under `Package settings`.)

## Moving to a new contract (contract bump)

`hack/capi/metadata.yaml` tracks the controller contract for each
releaseSeries. When cutting the first release of a new minor line whose
controller imports a different `sigs.k8s.io/cluster-api` contract,
prepend a new entry to the `releaseSeries` list **before** tagging:

```yaml
releaseSeries:
- major: 0
  minor: 3
  contract: v1beta3
- major: 0
  minor: 2
  contract: v1beta2
```

## Troubleshooting

### release.yml or release-helm.yml did not run after the tag push

Confirm you pushed the tag with a privileged identity (your normal git
credentials), not a CI-scoped `GITHUB_TOKEN`. Re-trigger manually:

```bash
gh workflow run release.yml      --field tag=vX.Y.Z --field dry_run=false
gh workflow run release-helm.yml --field tag=vX.Y.Z --field dry_run=false
```

### `release-helm.yml` times out in `wait-for-images`

`release.yml`'s `create-manifest` job has not finished yet (or failed).
Check the `release.yml` run and either wait for it to finish or fix the
underlying image build failure, then re-dispatch `release-helm.yml`.

### GHCR packages 404 with "name unknown"

The package was created private by default. Switch it to public with the
`gh api` snippet in the bootstrap section.

### `clusterctl init` rejects the provider with "contract not supported"

Either `metadata.yaml` declares the wrong contract or the installed
clusterctl is too old. Confirm with `clusterctl version` that you are on
`v1.12+` (v1beta2 contract) and that the contract line matches the minor
series you released.

### Need to re-push a chart with the same version

`release-helm.yml` is idempotent — it skips `helm push` when the target
version already exists on GHCR. To replace an existing chart you must
delete the OCI artifact from GHCR first (Package settings → Manage
versions → Delete). Bumping the tag is the safer alternative.
